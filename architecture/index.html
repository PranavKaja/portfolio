01 The CI/CD Pipeline (Git → Vercel)

// The Deployment Flow

There are no FTP uploads or manual server restarts. Vercel watches your GitHub repository. When you push code to the `main` branch, Vercel automatically deploys it.

1. Local Machine
You edit HTML/JS files in this repository.
→
2. Git Push
You run `git push origin main`. GitHub receives the code.
→
3. Vercel Webhook
Vercel is notified by GitHub. It pulls the code, optimizes assets, and deploys to a global CDN.
Why Vercel? Vercel provides a CDN (Content Delivery Network). When a user in Tokyo visits your site, Vercel serves the HTML files from a server in Tokyo, not New York. This makes load times nearly instantaneous. The vercel.json file in your root folder contains small configuration tweaks for how Vercel handles routing.

02 The File Structure Explained

Because there is no framework (like React or Next.js), the repository structure maps exactly to what the browser loads.

File/FolderRole
index.htmlThe homepage. It outlines the UI for the hero, the project grid, and the contact footer.
admin.html & admin.cssThe private dashboard UI. Includes the login screen and the tabbed interface for managing projects and skills.
style.cssThe global stylesheet. Contains all CSS variables (colors, fonts) and the core design system used across every page.
config.jsStores your public Supabase URL and anonymous key. Loaded on every page to initialize the database connection.
main.jsGlobal logic: custom cursor, mobile menu, layout animations, and utility functions used everywhere.
sections.jsFetches dynamic data for the public homepage (like the list of Skills or Projects) from Supabase.
admin.jsThe brains of the admin panel. Handles authentication, fetching secured data, the drag-and-drop skills canvas, and saving changes back to the database.
analytics.jsCustom tracking script. Silently logs pageviews and clicks to Supabase without using cookies.

03 The Database Connection (Supabase)

Supabase acts as your "Backend as a Service". It hosts a PostgreSQL database and wraps it in a REST API that your frontend JavaScript can call directly.

How the Browser talks to the Database

Every HTML file includes two critical scripts at the bottom:

  1. https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2 - The Supabase library.
  2. config.js - Your project keys.

Inside your JavaScript files (like sections.js), you simply write code like:
const { data } = await db.from('projects').select('*')

This translates into an HTTPS request sent directly from the user's browser to Supabase's servers, returning JSON data that your JavaScript then renders into the DOM.

Why isn't this dangerous?

The keys in config.js are public. Anyone can see them. So why can't a hacker delete all your projects?

Because of Row Level Security (RLS) inside Postgres.

When the browser sends a request using the public key, Supabase applies RLS policies. For the projects table, the policy says: "If the user is anonymous, only let them read rows where published=true. Do not let them update or delete."

When you log in via admin.html, Supabase gives your browser a secure JWT (JSON Web Token). When you make requests with that token, the database policy says: "This user is authenticated. Let them read, insert, update, and delete everything."
